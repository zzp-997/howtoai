## 一、项目背景

「极智协同」移动端企业 OA 办公应用已完成一期开发，现有功能包括：会议预定、差旅申请、请假管理、考勤打卡、文档查询、待办事项、公告通知、报销申请等模块。

一期已进行多轮迭代优化，包括：移动端删除对话框修复、登录页UI优化、智能功能（会议推荐、行程提醒）、FAB修复与页面过渡、首页优化等。

技术栈：
- 前端：Vue 3.5 + Vite + Pinia + TDesign Mobile + Vant 4 + Tailwind CSS
- 后端：Python Flask + MySQL
- 部署：GitHub Actions + Docker

## 二、二期规划目标

采用**分阶段推进**策略，共四个阶段：

| 阶段 | 主题 | 核心内容 | 预估周期 |
|------|------|----------|----------|
| 第一阶段 | 安全加固 | 登录保护、频率限制、操作日志、Token刷新、密码策略 | 1-2周 |
| 第二阶段 | 审批增强 | 多级审批、批量处理、消息通知、流程可视化、审批人配置、催办提醒、意见反馈 | 2-3周 |
| 第三阶段 | 数据可视化 | 会议报表、考勤报表、审批统计、综合大屏、知识库、会议智能调度 | 2-3周 |
| 第四阶段 | 任务协作 | 团队任务分配、进度跟踪、任务评论、子任务、协作通知 | 2周 |

### 新功能合并说明

| 新功能 | 合并至阶段 | 原因 |
|--------|------------|------|
| 意见反馈 | 第二阶段 | 共享消息通知机制，低复杂度快速产出 |
| 知识库 | 第三阶段 | 与报表同属"数据展示类"，补充文档模块 |
| 会议智能调度 | 第三阶段 | 与会议报表数据共享，合并为"会议模块增强" |
| 任务协作 | 第四阶段 | 复杂度最高，单独阶段专注开发 |

## 三、各阶段详细内容

### 第一阶段：安全加固

| 功能项 | 前端改动 | 后端改动 | 数据库改动 |
|--------|----------|----------|------------|
| 登录失败限制 | 登录页错误提示、锁定倒计时 | 失败计数、账户锁定逻辑 | login_logs 表 |

## 四、团队配置与角色职责

### 团队角色配置

| 角色 | 职责范围 | 参与阶段 |
|------|----------|----------|
| 产品经理 | 需求细化、原型设计、验收标准、用户调研 | 全阶段 |
| UI设计师 | 界面设计、交互规范、视觉规范、设计稿交付 | 全阶段 |
| 前端工程师 | Vue3 页面开发、图表集成、交互实现 | 全阶段 |
| 后端工程师 | Flask API 开发、安全机制、定时任务 | 全阶段 |
| 架构师 | 数据库设计、技术方案评审、性能优化、代码审查 | 全阶段 |
| 测试工程师 | 功能测试、安全测试、回归测试、性能测试 | 全阶段 |
| Team Leader | 自动驱动执行、任务分发、进度监控、阻塞处理 | 全阶段 |

### Team Leader 特殊职责

**核心理念：计划启动后，全流程自动化执行，无需人工介入确认下一步。**

| 能力 | 说明 |
|------|------|
| 自动任务分发 | 根据实施计划，自动识别当前阶段，将任务分发给对应角色智能体 |
| 依赖链管理 | 自动解析任务依赖关系，按序触发 |
| 进度实时监控 | 实时追踪各角色任务状态，更新进度看板 |
| 阻塞自动处理 | 检测到阻塞时，自动协调相关角色解决，或升级处理 |
| 阶段自动流转 | 当前阶段所有任务完成后，自动进入下一阶段，无需人工确认 |
| 结果自动验收 | 阶段完成后，自动触发测试验证，通过则流转，失败则回退修复 |

## 五、中断恢复机制

### 进度持久化方案

| 存储 | 内容 | 文件路径 |
|------|------|----------|
| 实施进度文件 | 当前阶段、当前任务、各角色状态 | `docs/task/phase2-progress.json` |
| 任务清单文件 | 完整任务列表、依赖关系、完成状态 | `docs/task/phase2-tasks.md` |
| 阶段产物目录 | 各阶段产出的设计稿、代码、测试报告 | `docs/task/phase2-artifacts/` |

### 进度数据结构

```json
{
  "version": "2.0",
  "project": "极智协同二期迭代",
  "currentPhase": 1,
  "currentStage": "安全加固",
  "status": "in_progress",
  "lastUpdate": "2026-04-30T12:00:00",
  "phases": [
    {
      "id": 1,
      "name": "安全加固",
      "status": "in_progress",
      "progress": 35,
      "tasks": [
        {"id": "1.1", "name": "数据库设计", "owner": "架构师", "status": "completed"},
        {"id": "1.2", "name": "登录保护开发", "owner": "后端工程师", "status": "in_progress"}
      ]
    }
  ],
  "team": {
    "产品经理": {"status": "idle", "currentTask": null},
    "前端工程师": {"status": "busy", "currentTask": "1.3"},
    "后端工程师": {"status": "busy", "currentTask": "1.2"}
  }
}
```

### 恢复机制

| 场景 | 处理方式 |
|------|----------|
| 正常中断 | 保存当前进度，标记各任务状态，下次读取后继续 |
| 阶段完成中断 | 标记阶段完成，下次直接进入下一阶段 |
| 任务执行中断 | 标记任务中断点，下次从该任务继续 |
| 异常中断 | 读取最后状态，回滚未完成的原子操作，重新执行当前任务 |

### 恢复指令

```
/phase2-continue   # 恢复二期迭代执行
/phase2-status     # 查看当前进度
/phase2-restart    # 重新开始（清空进度）
```

## 六、风险与缓解措施

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 多级审批流程复杂 | 开发周期延长 | 先实现单级审批增强，多级作为渐进迭代 |
| 消息通知实时性 | 用户体验 | 采用轮询 + WebSocket 可选方案 |
| 报表大数据量性能 | 页面响应慢 | 后端聚合 + 前端分页 + 缓存策略 |
| 任务协作权限复杂 | 开发难度高 | 参考 GitHub Issues 权限模型简化 |
| 中断恢复数据一致性 | 进度丢失 | 每个任务完成即持久化，关键节点校验 |

## 七、验收标准

### 整体验收

- [ ] 四个阶段全部功能开发完成
- [ ] 所有测试用例通过
- [ ] 无 P0/P1 级别 Bug
- [ ] 性能测试达标（接口响应 < 500ms，页面加载 < 3s）

### 分阶段验收

| 阶段 | 验收标准 |
|------|----------|
| 第一阶段 | 登录失败5次锁定15分钟；接口限流100次/分钟；操作日志完整记录 |
| 第二阶段 | 多级审批流程跑通；批量审批支持；消息实时推送；意见反馈可提交处理 |
| 第三阶段 | 各模块报表数据准确；大屏数据实时刷新；知识库可搜索；智能推荐有效 |
| 第四阶段 | 任务可分配多人；进度实时同步；评论支持@提醒；子任务进度联动 |
