## 一、项目概述

### 1.1 项目背景

极智协同是一款面向企业职场场景的 H5 移动端办公应用，旨在解决日常办公中的高频痛点，通过智能化功能设计，提升企业员工的办公效率和体验。

### 1.2 核心理念

- **深度契合职场真实诉求**：围绕会议预定、差旅出行、考勤打卡等高频场景进行产品设计
- **极致用户体验**：简洁直观的交互设计，减少用户认知负担
- **逻辑严密**：完整的数据流转和状态管理，确保业务逻辑准确
- **智能化升级**：通过数据分析和智能推荐，主动为用户提供决策支持

### 1.3 运行环境

- **运行平台**：H5 移动端网页，支持主流浏览器
- **数据存储**：完全基于 IndexedDB 本地存储
- **离线支持**：所有核心功能支持离线运行

### 1.4 用户角色

| 角色 | 说明 |
|------|------|
| 普通员工（用户端） | 日常办公操作：会议预定、差旅申请、请假打卡、文档查询、待办管理 |
| 管理员（管理员端） | 审批管理、会议室配置、文档管理、考勤管理、公告发布 |

## 三、核心功能模块设计

### 3.1 会议预定模块

#### 功能概述
解决企业会议室预定难、冲突多的痛点，提供可视化的预定管理和智能推荐功能。

#### 核心功能

| 功能 | 用户端 | 管理员端 |
|------|--------|----------|
| 会议室列表 | 查看名称、容量、设备 | 增删改会议室 |
| 预定日历 | 时间轴展示占用情况 | 查看所有预定 |
| 新建预定 | 表单+冲突检测 | 可删除任意预定 |
| 我的预定 | 列表+取消功能 | — |

#### 🌟 独特功能点：一键秒定空闲会议室

**设计思路**：
- 分析用户历史预定习惯，自动记录常用会议室和时段偏好
- 实时计算各会议室空闲状态，匹配用户需求（人数、设备）
- 首页直接展示"智能推荐"卡片，点击即可一键预定

**实现方式**：
```javascript
// 用户预定偏好记录
userMeetingPreferences: {
  frequentRooms: [roomId],        // 常用会议室
  frequentTimes: [{weekday, time}], // 常用时段
  lastBookingInfo: {...}          // 上次预定信息
}

// 智能推荐算法
1. 按历史使用频次排序会议室
2. 过滤当前时段空闲的会议室
3. 匹配人数和设备需求
4. 返回 Top 3 推荐结果
```

### 3.3 请假打卡模块

#### 功能概述
整合请假申请和每日考勤，提供智能提醒和补卡功能。

#### 核心功能

| 功能 | 用户端 | 管理员端 |
|------|--------|----------|
| 打卡操作 | 上班/下班打卡 | 考勤规则配置 |
| 打卡日历 | 月视图状态展示 | 全员考勤报表 |
| 请假申请 | 假别选择+余额显示 | 请假审批 |
| 补卡申请 | 忘记打卡后申请 | 补卡审批 |

#### 🌟 独特功能点：连休推荐与假期过期提醒

**设计思路**：
- 结合节假日配置，智能计算请假一天可获得的最大连休天数
- 检测年假余额和过期日期，主动提醒用户使用即将过期的假期
- 首页展示打卡状态卡片，进入时自动弹出打卡提醒

**实现方式**：
```javascript
// 连休推荐算法
1. 遍历未来30天的工作日
2. 检查前后是否有周末/节假日
3. 计算请假1天可获得的连休天数
4. 排序返回最优推荐

// 假期过期提醒
if (年假过期日期 - 今天 <= 60天 && 年假余额 > 0) {
  首页展示提醒卡片
}
```

| 文档操作 | 在线预览+下载 | Blob 存储 |

#### 🌟 独特功能点：智能搜索推荐

**设计思路**：
- 记录用户浏览历史，首页展示最近浏览和常用文档
- 搜索时自动联想历史搜索词和热门关键词
- 根据访问频率，自动排序"常用文档"

**实现方式**：
```javascript
// 文档浏览记录
documentViewLogs: {
  userId, documentId, viewedAt
}

// 搜索历史
searchHistories: {
  userId, keyword, searchedAt
}

// 热门搜索统计
SELECT keyword, COUNT(*) as count 
FROM search_histories 
GROUP BY keyword 
ORDER BY count DESC
```

#### 5.3.4 移动端适配方案

针对 H5 应用在真机上字体和图标过小的问题，采用以下适配方案：

**设计原理**：
- 使用 `postcss-px-to-viewport` 将 px 单位转换为 vw，实现响应式布局
- 设计稿基准宽度为 750px
- 通过 CSS 媒体查询，仅在移动端（屏幕宽度 ≤ 768px）应用整体放大

**核心实现**：

```css
/* 移动端专用放大样式 */
@media screen and (max-width: 768px) {
  #app {
    zoom: 1.2;  /* 移动端整体放大 20% */
  }
  /* 修复 zoom 导致的底部固定元素问题 */
  .fixed {
    zoom: 0.833;  /* 1 / 1.2 = 抵消放大 */
  }
}
```

**适配效果**：

| 设备类型 | 屏幕宽度 | zoom 值 | 显示效果 |
|---------|---------|---------|----------|
| PC 端 | > 768px | 1 | 原始大小，布局正常 |
| 移动端 | ≤ 768px | 1.2 | 整体放大 20%，字体清晰 |

**触摸优化**：

```css
/* 移除点击高亮 */
* {
  -webkit-tap-highlight-color: transparent;
}

/* 优化触摸滚动 */
html, body {
  -webkit-overflow-scrolling: touch;
  overscroll-behavior: none;
}

/* 防止 iOS Safari 自动调整字体 */
body {
  -webkit-text-size-adjust: 100%;
  text-size-adjust: 100%;
}
```

**设计变量系统**：

```css
:root {
  /* 字体大小系统 */
  --font-title-lg: 36px;    /* 大标题 */
  --font-title: 30px;       /* 标题 */
  --font-card-title: 28px;  /* 卡片标题 */
  --font-text-lg: 26px;     /* 大正文 */
  --font-text: 24px;        /* 正文 */
  --font-text-sm: 22px;     /* 小正文 */
  --font-desc: 20px;        /* 描述文字 */
  --font-small: 18px;       /* 最小文字 */

  /* 图标大小系统 */
  --icon-lg: 48px;          /* 大图标 */
  --icon-md: 32px;          /* 中图标 */
  --icon-sm: 28px;          /* 小图标 */
  --icon-xs: 24px;          /* 最小图标 */
}
```

**方案优势**：
- PC 端和移动端独立控制，互不影响
- 使用媒体查询，无需 JavaScript 判断
- 只需几行 CSS，代码简洁
- 兼容性好，支持主流移动浏览器

## 九、变更管理规范

### 9.1 OpenSpec 框架概述

本项目采用 **OpenSpec** 作为变更管理工具，规范化软件开发过程中的需求、设计和实现流程。所有后续迭代需求均需通过 OpenSpec 记录变更提案。

### 9.2 目录结构

```
openspec/
├── changes/              # 变更工件
│   ├── <change-name>/    # 具体变更目录
│   │   ├── proposal.md   # 变更提案（动机、范围）
│   │   ├── design.md     # 设计文档（技术方案）
│   │   └── tasks.md      # 任务清单（实施步骤）
│   └── archive/          # 已完成的变更归档
├── specs/                # 主规范文件
├── config.yaml           # OpenSpec 配置
└── README.md             # 使用说明
```

### 9.3 核心工作流

```
┌─────────────────────────────────────────────────────────────┐
│                    OpenSpec 工作流                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 探索 (/opsx:explore)                                    │
│     └─→ 分析问题、探索方案、可视化设计                        │
│                                                             │
│  2. 提议 (/opsx:propose)                                    │
│     └─→ 创建 proposal.md、design.md、tasks.md               │
│                                                             │
│  3. 实施 (/opsx:apply)                                      │
│     └─→ 按 tasks.md 逐步实现，标记任务完成状态                │
│                                                             │
│  4. 归档 (/opsx:archive)                                    │
│     └─→ 完成后归档变更，更新主规范                            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 9.4 变更命名规范

使用 kebab-case 命名变更，格式：`<action>-<target>-<detail>`

| 类型 | 示例 |
|------|------|
| 新增功能 | `add-user-authentication` |
| 修复问题 | `fix-mobile-delete-dialog` |
| 重构优化 | `refactor-data-layer` |
| 更新文档 | `update-api-documentation` |

### 9.5 工件说明

| 工件 | 用途 | 内容要求 |
|------|------|----------|
| `proposal.md` | 变更提案 | 背景、动机、范围、非目标、风险评估 |
| `design.md` | 设计文档 | 技术方案、架构决策、代码模式、测试策略 |
| `tasks.md` | 任务清单 | 可执行步骤、完成状态、验证项 |

### 9.6 最佳实践

1. **分阶段执行**：不跳过探索阶段直接实施
2. **工件完整**：确保 proposal、design、tasks 三件套齐全
3. **最小变更**：每个任务代码变更保持最小化
4. **状态同步**：完成后及时标记任务状态
5. **及时归档**：所有任务完成后执行归档
